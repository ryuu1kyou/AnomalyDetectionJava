# トレーサビリティ 1枚チェックリスト（プロジェクト版）

## 使い方

開発アイテムごとに最低限の抜け漏れを防ぐ確認用。全項目を毎回埋める必要はない。
まず「リーダ向け確認3問」だけでも効果がある。

Feature: \_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_
Decision: \_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_
Change: \_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_
Reviewer: \_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_
Date: \_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_

---

## リーダ向け確認3問（優先）

- [ ] この変更は何のまとまりか → `feature_id`
- [ ] IF に意味変更が入っていないか → `if_version` / `signalMeaning`
- [ ] unknown が放置されていないか → `unknownDeadline`

---

## 1. 変更の起点

- [ ] `feature_id` がある
- [ ] `decision_id` または DR 履歴がある
- [ ] 要求ID / 案件名が分かる

## 2. モデルと IF

- [ ] `module_id` が分かる
- [ ] IF 変更有無が分かる
- [ ] IF 変更がある場合、`if_version` が更新されている
- [ ] `changeReason` が記録されている
- [ ] 意味変更（出力条件・解釈ロジック）を見落としていない

## 3. 実装と接続

- [ ] 対応する git commit hash が分かる
- [ ] `feature_id` `decision_id` `module_id` `commit` がつながる
- [ ] `change_id` がある（change 単位で追う場合）

## 4. OEM / 車両 / 適用範囲

- [ ] 共通変更か OEM 固有変更か分かる
- [ ] 対象 OEM が分かる
- [ ] `UNKNOWN` のままの項目が明示されている
- [ ] `unknownDeadline` が設定されている

## 5. 台帳反映

- [ ] 承認台帳（OemApproval）に必要情報が入っている
- [ ] 実装接続台帳（ImplementationConnection）に必要情報が入っている
- [ ] 設計意図台帳（DesignIntent）に残すべき判断が記録されている

## 6. 最終確認

- [ ] 他人が見ても変更の起点と反映先を追える
- [ ] 担当者の頭の中だけに残っている判断がない
- [ ] 後から再説明できる最低限の情報が残っている
- [ ] 自分以外の誰かがこれを見て10分以内に説明できる

---

## 導入初期で無理にやらなくてよいこと

- [ ] 全エンティティの棚卸しを一気にやる
- [ ] 過去履歴を完全復元する
- [ ] すべてのテーブルを最初から完璧に設計する

上の3つは「やらない」と決めてよい。
まずはこれから入る変更に対して最低限の接続を残すことを優先する。
